# llmreview rules — one rule per line
# These rules guide the LLM during code review.
# Examples:
# All public functions must have KDoc comments.
# Avoid using !! (non-null assertion) in Kotlin.
# Error messages must be user-friendly and actionable.
no star/wildcard imports
no inline comments unless absolutely necessary
only use inline comments to explain why something is done, never what is being done
prefer functional style over imperative
no single line or bracketless if statements
prefer `it` over named lambda parameters
